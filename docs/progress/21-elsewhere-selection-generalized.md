# 21 — elsewhere-selection-generalized

사용자 질문: "Slack뿐 아니라 저런 형태 메신저나 뭐든 다 같은 오류 날 텐데 같이
잡았나?" — 20의 수정은 웹 엔진 계열(AXWebArea 보유: Electron 전부 + 브라우저)만
커버. 네이티브 메신저(카카오톡·Messages 등)는 웹 영역이 없어 미커버였다.
실측: 카카오톡은 포커스 요소가 아예 NIL — 동일 증상(선택 누락 → 삽입 모드 →
입력칸에 결과) 예상.

## 일반화 설계 — 신호 기반 분기 (captureContext)

포커스 요소의 자기 선택이 빌 때:

1. 범위 > 0 → ⌘C, 요소 소속 선택 (기존).
2. **웹 영역 조상 있음 → 웹 영역이 권위**: 문서 선택 있으면 보기 전용(20),
   없으면 그대로 믿고 끝 — 여기서 ⌘C를 쏘면 줄 복사 에디터(VS Code 등: 빈 선택
   ⌘C = 현재 줄 복사)의 오탐이 부활한다. 부수 효과로 웹 앱 빈 입력칸의 기존
   ⌘C(0.3s)도 사라져 핫키가 빨라짐.
3. **웹 영역 없음(네이티브) + full 읽힘 → 투기적 ⌘C(0.15s)**: 텍스트가 나왔고
   **요소 자기 내용의 일부가 아니면**(줄 복사 오탐 배제) 앱 내 다른 곳의 선택 =
   보기 전용, 앵커 nil(마우스 폴백). 요소 내용의 일부면 phantom — 폐기.
4. full == nil (AX 침묵) → ⌘C, 요소 소속 (레거시 유지 — 요소 상태를 알 길 없음).
5. **포커스 요소 NIL(카카오톡류) → ⌘C(0.15s)**: 텍스트 나오면 위치 미상의 선택 =
   보기 전용 (앵커 nil). 아니면 기존대로 편집 가능 삽입 컨텍스트.

트레이드오프 (의도):
- 네이티브 삽입 핫키(포커스 입력칸, 선택 없음)에 ⌘C 타임아웃 최대 0.15s 지연 추가.
- 네이티브에서 선택이 입력칸 초안의 부분 문자열과 우연히 일치하면 phantom으로
  오폐기 → 삽입 모드 (구 동작과 동일한 누락 — 안전한 방향).
- ⌘C가 앱 내 선택에 안 닿는 네이티브 앱(첫 응답자가 입력칸이고 선택 뷰가 복사
  체인에 없음)은 여전히 누락 — 신호가 없어 잡을 수 없음, 사례 나오면 재검토.
